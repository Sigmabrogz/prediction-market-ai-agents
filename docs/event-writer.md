# Event Writer Architecture

The `EventWriter` service is the central persistence boundary of the Prediction Market AI platform. It is entirely decoupled from the execution loop. 

## Responsibilities
1. Subscribe to Redis `oracle_signals` and `order_lifecycle_events`.
2. Maintain strict idempotency.
3. Translate raw JSON envelopes into strict Postgres rows.
4. Run background aggregations (e.g. updating Positions based on Fills).

## Idempotency Rules
Distributed systems over pub/sub are prone to "at least once" delivery anomalies. The EventWriter protects the DB via:
1. **Primary Keys:** Every `Signal` and `OrderLifecycleEvent` uses the exact UUID generated by the publisher as its database `id`. 
2. **Pre-flight Lookups:** Before executing `prisma.create()`, the writer checks `findUnique`. If the record exists, the event is silently dropped to prevent duplicate processing.
3. **Fill Determinism:** Simulated/Live fill rows are generated with a deterministic composite string (`fill-{orderId}-{timestamp}-{size}`) to guarantee the exact same slice of liquidity isn't credited to a position twice.

## Data Flow
- `SIGNAL_RECEIVED` -> Creates the `Order` skeleton using a surrogate ID.
- `ORDER_SUBMITTED` -> Upserts the `Order` with the real Exchange ID.
- `ORDER_FILLED` -> Triggers the `handleFillEvents()` -> Computes VWAP -> Upserts `Position`.
